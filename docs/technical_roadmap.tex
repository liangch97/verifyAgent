\begin{abstract}\noindent
本文描述 \texttt{contract-review-openclaw} 项目的整体技术路线：从一份合同
\texttt{.docx} 到行政老师 IM 中收到的 \emph{行政版 PDF 报告} 的端到端管线，
包括 OpenClaw skill 体系、入口编排、合同读取与字段抽取、规则引擎、
学校范本对照、企业工商核验、报告渲染与 IM 交付、数据契约与部署形态。
文档读者：项目维护者、二开者、运维。
\end{abstract}

\section{项目概述}

\subsection{背景与目标}

中山大学每年签订大量横向科研合同（技术开发、技术服务、技术咨询、技术转让、
共同申请专利、共同申请专利等），这些合同需要由行政老师做\emph{形式化审核}：
检查合同要素是否齐全、关键条款是否合规、知识产权归属是否清楚、甲方企业是否
真实存在等。该过程长期依赖人工逐条比对，工作量大、容易疏漏。

\textbf{contract-review-openclaw} 是面向 OpenClaw 平台的本地包，目标是：

\begin{enumerate}[leftmargin=1.4em,itemsep=2pt]
  \item \textbf{IM 一键审核}。行政老师在飞书里发一份合同 + 一句话「请帮我
    审核这个合同」，自动返回一份可直接转发给项目负责人的\emph{行政版 PDF}。
  \item \textbf{形式化覆盖}。覆盖 41 条形式化规则，区分自动判定项 / 人工
    复核项 / 仅留档项。
  \item \textbf{学校范本对照}。识别合同是否基于中山大学合同范本起草，
    并指出条款级偏离。
  \item \textbf{企业工商核验}。通过企查查（QCC）扫码登录获取甲方真实工商
    信息，与合同抬头比对；无法核验时降级为人工材料路径。
  \item \textbf{本地优先 / 离线可用}。所有审核逻辑跑在本地，不依赖外部
    LLM；企查查访问全程由用户控制。
\end{enumerate}

\subsection{用户与角色}

\begin{longtable}{p{2.6cm} p{12.8cm}}
\toprule
\textbf{角色} & \textbf{典型动作} \\ \midrule
\endhead
\textbf{行政老师} & 在飞书里发合同 + 触发指令；阅读最终 PDF；转发给项目
负责人 / 合作方修改。\textbf{不需要}理解 JSON、规则编号或脚本。 \\
\textbf{二开 / 测试} & 在 PowerShell / Bash 里跑 \texttt{run\_contract\_review.py}
做批量回归；维护 \texttt{horizontal\_contract\_formal\_rules.yaml}；
新增范本时复制 \texttt{.docx} 到 \texttt{references/templates/}。 \\
\textbf{运维} & 部署 OpenClaw / WSL；维护飞书凭据；监控
\texttt{output/im\_review/<timestamp>/} 的实际推送。 \\
\bottomrule
\end{longtable}

\subsection{安全与边界}

\begin{itemize}[leftmargin=1.4em,itemsep=2pt]
  \item \textbf{不调用外部 LLM}。所有规则匹配、字段抽取、范本对比均为
    本地代码（正则 + \texttt{python-docx} + \texttt{difflib}）。
  \item \textbf{不绕过任何访问控制}。企查查走标准浏览器访问；遇到验证码、
    WAF、登录失败、付费墙时主动降级到「人工材料 / 退回普通审核」。
  \item \textbf{企业信息白名单}。甲方信息只能来自三种受控来源：本次 QCC
    扫码、人工 \texttt{--company-check} 文件、用户提供的截图 / 记录。
    不允许从模型记忆或公开知识填充。
  \item \textbf{结果可解释}。每条 finding 必须给出 \texttt{rule\_id} +
    \texttt{evidence} + \texttt{location}；最终面向老师的 PDF 隐去技术名词，
    按「问题 / 关联条款 / 建议处理」三列展示。
\end{itemize}

\section{入口层：三种触发方式}

\subsection{IM（飞书）：默认触发}

行政老师对 OpenClaw 说：

\begin{lstlisting}[style=shell]
请帮我审核这个合同，<合同文件完整路径>
\end{lstlisting}

OpenClaw 触发 \texttt{contract-formal-review-flow} skill；该 skill 强制要求
模型只走一条命令：

\begin{lstlisting}[style=shell]
python3 /root/full_qcc_review.py "<contract_file_path>"
\end{lstlisting}

\subsubsection{\texttt{full\_qcc\_review.py} 编排器}

外置在 OpenClaw 部署目录的包装器（691 行）。它的责任：

\begin{enumerate}[leftmargin=1.4em,itemsep=2pt]
  \item \textbf{初步推送状态}。\texttt{fs\_text("📋 开始审核\textbackslash n
    合同: ... 步骤 1/4: 启动 QCC 登录...")}
  \item \textbf{快速路径检测}。如果检测到飞书凭据缺失或合同名像
    \emph{已审核报告}（避免对生成的 PDF 再做审核），直接走「无 QCC」
    路径，跳到第 4 步。
  \item \textbf{抽取党事方甲方}。\texttt{extract\_party\_a()} 用正则从合同
    抽出甲方名字；如果用户传了 \texttt{--company-name}，以参数为准。
  \item \textbf{QCC 扫码登录}。复用 \texttt{demos/qcc\_login\_demo.py} 的
    Playwright 浏览器；优先复用 \texttt{.browser-profiles/qcc-demo} 的
    本机会话；失效时通过 \texttt{fs\_image} 把二维码截图推到飞书让老师
    扫码；最长等 15 分钟。
  \item \textbf{QCC 公司搜索}。登录成功后搜索甲方公司，截屏，导出
    \texttt{output/qcc\_login\_demo/company\_check\_draft\_*.json}。
  \item \textbf{触发本地审核}。\\
    \texttt{python run\_contract\_review.py "<contract>" --company-check
    "<draft.json>"}\\
    生成 Markdown / JSON / Excel 三件套到 \texttt{output/}。
  \item \textbf{渲染行政 PDF}。导入 \texttt{admin\_report.py}，把
    \texttt{contract\_findings\_<hash>.json} + Markdown 渲染成
    \texttt{合同审核报告\_行政版.pdf}（含「三、学校范本对照」一节）。
  \item \textbf{推送 PDF}。\texttt{fs\_file()} 通过 openclaw-gateway 把
    PDF 上传到飞书并发给老师。
  \item \textbf{回归归档}。所有产物落到
    \texttt{output/im\_review/<timestamp>/} 便于回查。
\end{enumerate}

\textbf{失败路径}：QCC 验证码 / 405 / 403 / 登录超时 / 浏览器崩溃 / 飞书
凭据缺失，全部包装为「降级到无 QCC 审核」并在最终 PDF 里写明
\emph{「外部企业核验未完成，请补充企查查截图或人工核验记录」}。

\subsection{命令行：\texttt{run\_contract\_review.py}}

\begin{lstlisting}[style=shell]
# 仅本地审核（不查 QCC）
python run_contract_review.py "<contract.docx>"

# 带人工整理的企业核验材料
python run_contract_review.py "<contract.docx>" \
       --company-check "<company_check.json|csv|xlsx>" \
       --rules <rules.yaml>          # 默认 references/rules/horizontal_*.yaml
       --output-dir <dir>            # 默认 ./output
\end{lstlisting}

[md]   ./output/contract_review.md
[json] ./output/contract_findings_<hash>.json
[xlsx] ./output/合同形式化审核总表.xlsx
\end{lstlisting}

\section{OpenClaw skill 体系}

\subsection{两个 skill 的分工}

OpenClaw 在解析用户消息时按 skill description 中的触发词激活相应 skill。
本项目当前提供 \emph{两个并列} skill：

\begin{longtable}{p{4.6cm} p{2.8cm} p{7.5cm}}
\toprule
\textbf{Skill} & \textbf{规模} & \textbf{触发词与职责} \\ \midrule
\endhead
\texttt{contract-formal-review-flow} & 11.4 KB / 214 行
& 触发：\emph{合同审核 / 横向合同 / 帮我审核 / 审核这个合同 /
SYSU contract review}。强制模型只走 \texttt{full\_qcc\_review.py}，
保证 IM 模式下\textbf{不输出文本摘要}（PDF 即交付）。 \\
\texttt{contract-template-compliance} & 7.2 KB / 99 行
& 触发：\emph{范本 / 范式 / 示范文本 / 学校范本 / 范本对照 /
模板对照 / 条款偏离}。读取 \texttt{template\_match} JSON 字段并产出
范本符合性结论；不重复执行任何脚本。 \\
\bottomrule
\end{longtable}

\subsection{为何要拆 skill}

Anthropic Skills 设计准则之一：\emph{每个 SKILL.md 单一职责，体量 $\le$ 5 KB}。
当一个 skill 承担多个异构工作流时，模型 attention 被稀释，会出现「忘记
调用工具」「混淆 IM 模式回复结构」等表现。本项目早期把范本对照塞进
主 skill 后，主 SKILL.md 涨到 22 KB / 277 行（覆盖 7 类工作流），随即出现
skill 把握能力下降；v1.1 拆分后恢复正常。

\subsection{安装机制}

\begin{lstlisting}[style=py]
# install_into_openclaw.py 的核心逻辑
SOURCE_SKILLS = PROJECT_ROOT / "skills"
TARGET = Path.home() / ".openclaw" / "skills"
for skill_dir in SOURCE_SKILLS.iterdir():
    target_dir = TARGET / skill_dir.name
    shutil.copytree(skill_dir, target_dir, dirs_exist_ok=True)
    # 把 SKILL.md 里的 <PROJECT_ROOT> 占位符替换为本机绝对路径
    skill_md = target_dir / "SKILL.md"
    skill_md.write_text(
        skill_md.read_text().replace("<PROJECT_ROOT>", str(PROJECT_ROOT))
    )
\end{lstlisting}

\section{学校范本对照子系统}

\subsection{背景：单门槛失败案例}

v1.0 的 \texttt{template\_matcher.py} 只用 \texttt{difflib.SequenceMatcher}
对「合同正文 vs 范本正文（去波动 token 后）」算相似度，超过
$\theta=0.40$ 即判定为「基于学校范本」。失败案例：行业起草的技术服务
合同（乙方为中山大学），实测相似度 $\approx 0.56$，被误判为基于
\texttt{中山大学技术服务合同.docx}；进一步使「使用说明」段被切成
「条款」，导致行政 PDF 第三节冒出 28 个 \texttt{removed} + 32 个
\texttt{added} 噪声。

\subsection{双门槛设计}

新版判定：\textbf{形状信号与内容信号同时通过}：

\begin{equation}\label{eq:gate}
\text{matched} \;=\;
\bigl(s_\text{best} \ge \theta_\text{shape}\bigr)
\;\wedge\;
\bigl(\#\text{anchor}_\text{contract} \ge k_\text{anchor}\bigr)
\end{equation}

\begin{itemize}[leftmargin=1.4em,itemsep=1pt]
  \item $s_\text{best}=\max_T \text{Ratio}\bigl(\sigma(C),\sigma(T)\bigr)$，
    $C$ 是合同文本、$T$ 是范本文本、$\sigma$ 是
    \texttt{\_strip\_volatile}（清掉日期 / 金额 / 占位符等易变 token）。
  \item $\theta_\text{shape}=0.65$。
  \item \texttt{anchor} 集为\textbf{学校范本独有}的短语集合（见
    表~\ref{tab:anchors}），$k_\text{anchor}=1$。
\end{itemize}

\begin{table}[h]
\centering
\caption{SYSU 范本独有锚点短语清单}\label{tab:anchors}
\begin{tabular}{p{8.0cm} p{6.0cm}}
\toprule
\textbf{保留（独有）} & \textbf{已剔除（中性 / 误导）} \\ \midrule
\texttt{【科学研究院】} & \texttt{中山大学} \\
\texttt{【科研管理部门】} & \texttt{中大} \\
\texttt{学校合同管理信息系统} & \texttt{示范文本} \\
\texttt{合同管理信息系统} & \\
\texttt{中山大学·深圳} / \texttt{中山大学•深圳} / \texttt{中山大学∙深圳} & \\
\texttt{本合同由【科学研究院】} & \\
\texttt{本合同由【科研管理部门】} & \\
\bottomrule
\end{tabular}
\end{table}

\textbf{为什么剔除 \texttt{中山大学} / \texttt{示范文本} / \texttt{中大}}：
中大经常作为乙方出现在行业自拟合同中（这只能说明合同的乙方是中大，并不
意味着合同采用了\emph{学校范本}）；\texttt{示范文本} 还会出现在国家科技部
范本里。这三个短语没有判别力。

\subsection{使用说明剥离}

SYSU 范本 \texttt{.docx} 起始有「使用说明」段（\texttt{一、二、三、…七、}）。
\texttt{\_strip\_preamble} 在文本中存在 \texttt{第 N 条} 时，丢弃该处
之前的所有行；否则保留原文。

\subsection{Clauses 级 diff}

匹配通过后做条款级对比：

\begin{enumerate}[leftmargin=1.4em,itemsep=1pt]
  \item \texttt{\_split\_clauses}：按 \texttt{HEADING\_RE} 切条款。
  \item \texttt{\_match\_clause\_pairs}：在范本条款与合同条款之间做
    匈牙利式对齐（按 \texttt{difflib.ratio} 最大化）。
  \item 标注 5 状态：\texttt{unchanged / modified / rewritten /
    added / removed}。
  \item 输出 \texttt{counts} 统计 + 每条条款的 \texttt{template\_excerpt}
    与 \texttt{contract\_excerpt}（最多 120 字）。
\end{enumerate}

\subsection{范本库}

\texttt{references/templates/} 收纳学校发布的合同范本：

\begin{lstlisting}[style=shell]
中山大学专利技术转让合同.docx
中山大学共同申请专利合同.docx
中山大学技术咨询合同.docx

中山大学技术服务委托合同.docx

\subsection{QCC 流程（\texttt{demos/qcc\_login\_demo.py}）}

\begin{enumerate}[leftmargin=1.4em,itemsep=1pt]
  \item 启动 Playwright Chromium，复用本机 \texttt{.browser-profiles/qcc-demo}。
  \item 打开 \texttt{https://www.qcc.com}，点击「登录」。
  \item 截二维码 → 通过 \texttt{full\_qcc\_review.fs\_image} 推到飞书。
  \item 轮询登录状态；超时（默认 15 分钟）后退出。
  \item 登录成功后搜索甲方公司，截取详情页。
  \item 解析详情页文字（基本信息、统一信用代码、法人、地址、状态、
    关联关系、风险信息），写入
    \texttt{output/qcc\_login\_demo/company\_check\_draft\_*.json}。
\end{enumerate}

\subsection{核验匹配（\texttt{company\_check\_matcher.py}）}

读取 \texttt{company\_check} JSON 并与合同抽取出的甲方做匹配：

\begin{itemize}[leftmargin=1.4em,itemsep=1pt]
  \item \texttt{party\_a\_match}：完全匹配 / 字面差异（如全角半角）/
    完全不一致。
  \item \texttt{related\_party\_check}：是否与中山大学存在关联（公开
    股权 / 校办企业 / 校友企业声明）。
  \item \texttt{military\_defense\_check}：检索是否有军工 / 涉密属性。
\end{itemize}

输出落到 \texttt{review.external\_checks}，参与
\texttt{contract\_rule\_checker.py} 的最终 finding 与决策。

\subsection{失败降级路径}

\begin{itemize}[leftmargin=1.4em,itemsep=1pt]
  \item QCC 登录超时 / 验证码 / 405 / 403 → 降级为「无 QCC」审核，
    \texttt{external\_checks.source = "manual"}，并在最终 PDF 写明
    「外部企业核验未完成」。
  \item 飞书凭据缺失（\texttt{[config] Feishu APP\_ID/...}）→
    跳过 IM 推送，但其它产物正常落到 \texttt{output/}。
  \item 合同名疑似「已审核报告」（\texttt{looks\_like\_previous\_report}）
    → 不再触发 QCC 与重审，避免循环。
\end{itemize}

\section{后续路线}

\begin{enumerate}[leftmargin=1.4em,itemsep=2pt]
  \item \textbf{语义化范本匹配}。在 \texttt{template\_matcher} 中加入
    条款级 embedding（如 BGE-small-zh）做语义相似度，与现有形状信号
    融合，进一步降低边缘案例（相似度 0.55$\sim$0.65）的误判风险。
  \item \textbf{条款级修改建议}。把 \texttt{clauses[].status=rewritten}
    的条款交给规则引擎做二次审查，生成针对性的修改建议（而不仅是
    「重大改写需复核法务」一句话）。
  \item \textbf{多范本复合识别}。支持「主范本 + 附加条款范本」的复合
    识别（如「专利技术转让 + 共同申请专利」）。
  \item \textbf{规则 YAML 编辑器}。给二开提供轻量 Web 编辑器，
    校验 \texttt{scope} 与 \texttt{check\_type} 的合法组合，避免误改。
  \item \textbf{Skill 元数据预检}。\texttt{install\_into\_openclaw.py}
    增加 YAML 解析 + \texttt{description} 长度 / 触发词覆盖度的预检，
    避免装入破损的 \texttt{SKILL.md}。
  \item \textbf{合成回归集}。构造 30+ 条「微改写 / 重命名甲乙方 /
    调整条款顺序」的合成样本，确保 \texttt{counts} 在改动后仍稳定。
  \item \textbf{运行时观测}。把 \texttt{full\_qcc\_review.py} 的步骤
    时间、QCC 成功率、Feishu 推送成功率上报到一个本地仪表盘。
\end{enumerate}
